# Structure Explanation: Problem Bundles and Experiment Runner

This document explains the structure of the `bundles[]` dictionary generated by `problemBuilder()` and the `run_experiment()` function flow.

## 1. Structure of `bundles[]` from `problemBuilder()`

The `build_problems_for_slab_type(slab_type)` function returns a dictionary called `bundles` with the following structure:

### Top-Level Structure

```python
bundles: dict[str, dict]
```

- **Key**: `problem_ID` (string) - e.g., `"1"`, `"2"`, `"3"`, etc.
  - Each problem ID corresponds to one row in the `problem_list.csv` file for the given slab type
- **Value**: A dictionary containing all problem-specific information (see below)

### Bundle Dictionary Structure

Each bundle (value in the dictionary) contains the following keys:

```python
{
    "problem": IOHProblem,                    # IOH experimenter problem object
    "var_names": list[str],                   # List of variable names (e.g., ["t_mm", "span_m"])
    "constraints": dict[str, dict],           # Dictionary of constraint configurations
    "decode": callable,                       # Decode function: decode(x_idx) -> params dict
    "label": str,                             # Human-readable problem label/description
    "ctx": EvalContext,                       # Evaluation context with caching
    "active_constraint_names": list[str]      # List of active constraint names
}
```

**Key observations:**
- Each problem ID maps to a complete bundle with all necessary components
- Different problems can have different numbers of variables (problem "1" has 2 vars, problem "2" has 3 vars)
- The `decode` function is problem-specific and knows how to map indices to parameter values
- The `ctx` (EvalContext) caches analysis results to avoid redundant computations
- Only active constraints appear in `active_constraint_names` (e.g., "crack_width_limit" is inactive in problem "1")

### Detailed Field Descriptions

#### `"problem"`: IOHProblem Object
- An IOH (IOHexperimenter) problem wrapper
- Callable: `problem(x_idx)` where `x_idx` is a list of integer indices
- Returns: penalized objective value `y_p` (float)
- Contains:
  - `problem.meta_data.n_variables`: number of variables (dimension)
  - `problem.bounds.lb`: lower bounds (list of integers)
  - `problem.bounds.ub`: upper bounds (list of integers)
  - `problem.state.current_best`: best solution found so far
    - `.x`: best design vector (list of integer indices)
    - `.y`: best objective value (float)

#### `"var_names"`: list[str]
- Example: `["t_mm", "span_m", "reinforcement_type"]`
- Ordered list matching the dimension of the problem
- Each name corresponds to one optimization variable

#### `"constraints"`: dict[str, dict]
- Dictionary mapping constraint names to their configurations
- Example structure:
  ```python
  {
      "deflection_limit": {
          "active": True,
          "enforced": CE.HIDDEN,  # ConstraintEnforcement enum
          "weight": 1.0,
          "exponent": 1.0,
          "kind": "real"
      },
      "strength_limit": {
          "active": True,
          "enforced": CE.HIDDEN,
          "weight": 1.0,
          "exponent": 1.0,
          "kind": "real"
      }
  }
  ```

#### `"decode"`: callable
- Function signature: `decode(x_idx: list[int]) -> dict[str, Any]`
- Converts integer index vector `x_idx` to a parameter dictionary
- Example: `decode([2, 4])` → `{"t_mm": 200, "span_m": 5.0}`

#### `"label"`: str
- Human-readable description of the problem
- Example: `"Height variation - fixed cost and CO2"`
- Used for logging and result organization

#### `"ctx"`: EvalContext Object
- Evaluation context that caches analysis results
- Key methods:
  - `ctx.get(x_idx)`: Returns analysis results dict for given design vector
    - Caches results to avoid redundant analysis calls
    - Returns: `{"y": float, "y_p": float, "violations": dict, "params": dict, ...}`
  - `ctx.ensure_constraints(names)`: Declares constraint attributes for logging
  - `ctx.ensure_params(var_names)`: Declares parameter attributes for logging
  - `ctx.reset()`: Clears cache and resets hit/miss counters
- Public attributes (set during `get()`):
  - `ctx.y`: Unpenalized objective value
  - `ctx.y_p`: Penalized objective value
  - `ctx.hits`: Cache hits count
  - `ctx.misses`: Cache misses count
  - `ctx.c__<constraint_name>`: Raw violation value for each constraint
  - `ctx.var__<var_name>`: Decoded parameter value
  - `ctx.idx__<var_name>`: Original index value

### How Algorithms Access the Analysis Function

The connection between optimization algorithms (like TPE) and your analysis function happens through a chain of function calls:

#### Call Chain Flow

```
Algorithm (TPE/DIRECT/etc.)
    ↓ calls problem(x_idx)
IOHProblem wrapper
    ↓ calls objective_idx(x_idx)
objective_idx function (in problem_builder.py)
    ↓ calls ctx.get(x_idx)
EvalContext.get()
    ↓ calls decode(x_idx) → params dict
    ↓ calls analysis_fn(params) → results dict
Slab-specific analysis function (e.g., analysis_ec2_de.analysis)
    ↓ performs structural analysis
    ↓ returns {"y": ..., "y_p": ..., "violations": {...}, ...}
```

#### Detailed Step-by-Step

1. **Algorithm calls the IOH problem**:
   ```python
   # Inside HyperoptTPESearch.__call__() or similar
   x_int = [2, 4]  # Design vector as integer indices
   y = problem(x_int)  # Calls IOHProblem
   ```

2. **IOHProblem wraps `objective_idx` function**:
   ```python
   # In problem_builder.py, line 191-199
   p = ioh.wrap_problem(
       function=objective_idx,  # This function is wrapped
       ...
   )
   ```

3. **`objective_idx` extracts penalized objective from context**:
   ```python
   # In problem_builder.py, line 176-188
   def objective_idx(x, _ctx=ctx, _n=n_vars, _problem_ID=problem_ID):
       rec = _ctx.get(x)  # Get analysis results (cached or computed)
       return float(rec.get("y_p", float("inf")))  # Return penalized objective
   ```

4. **`EvalContext.get()` manages caching and calls analysis**:
   ```python
   # In problem_evaluator.py, line 39-86
   def get(self, x_idx):
       key = tuple(int(k) for k in x_idx)
       if key == self.last_key:
           # Cache hit: reuse previous result
           self.hits += 1
           rec = self.last_data
       else:
           # Cache miss: compute new result
           self.misses += 1
           params = self.decode(x_idx)  # Convert indices to parameter dict
           rec = self.results(params)   # Call analysis function
           self.last_key = key
           self.last_data = rec
       # ... publish attributes for logging ...
       return rec
   ```

5. **Analysis function is bound during problem building**:
   ```python
   # In problem_builder.py, line 144-146
   analysis_fn = partial(slab_module.analysis, constraints=constraints_log)
   ctx = EvalContext(decode, analysis_fn)
   ```
   - `analysis_fn` is a partial function that binds the constraint configuration
   - When called with `params`, it calls `slab_module.analysis(params, constraints=constraints_log)`

6. **Slab-specific analysis function performs the actual computation**:
   ```python
   # Example: analysis_ec2_de.py, line 85
   def analysis(params: dict, constraints: dict) -> dict:
       # Performs structural analysis using params
       # Computes objective values (y, y_p)
       # Checks constraints and computes violations
       return {
           "y": unpenalized_objective,
           "y_p": penalized_objective,
           "violations": {...},
           "params": params,
           ...
       }
   ```

#### Key Points

- **Caching**: `EvalContext` caches results to avoid redundant analysis calls when the same design vector is evaluated multiple times
- **Decoding**: Integer indices (`x_idx`) are converted to actual parameter values (`params`) before analysis
- **Separation of concerns**: 
  - Algorithms only see integer indices and objective values
  - IOHProblem handles the optimization interface
  - EvalContext manages caching and coordinates decode/analysis
  - Analysis function performs domain-specific computations
- **Constraint handling**: Constraints are bound to the analysis function via `partial()`, so the analysis knows which constraints to check and return

#### Example Trace

```python
# Algorithm (TPE) wants to evaluate design [2, 4]
x_idx = [2, 4]

# 1. Algorithm calls problem
objective_value = problem([2, 4])

# 2. IOHProblem calls objective_idx([2, 4])
#    → objective_idx calls ctx.get([2, 4])

# 3. EvalContext.get([2, 4]) checks cache
#    → Cache miss → calls decode([2, 4])
#    → decode returns {"t_mm": 200, "span_m": 5.0}
#    → calls analysis_fn({"t_mm": 200, "span_m": 5.0})

# 4. analysis_fn calls analysis(params={"t_mm": 200, "span_m": 5.0}, constraints={...})
#    → Performs structural analysis
#    → Returns {"y": 150.5, "y_p": 152.3, "violations": {...}, ...}

# 5. EvalContext caches result and returns it
#    → objective_idx extracts y_p = 152.3
#    → Returns to algorithm

# If algorithm evaluates [2, 4] again:
#    → Cache hit → returns cached result immediately (no analysis call)
```

#### `"active_constraint_names"`: list[str]
- Example: `["deflection_limit", "strength_limit", "crack_width_limit"]`
- List of constraint names that are active (not disabled) for this problem
- Used for logging and constraint enforcement

### Example Usage

```python
from slabdesignbench.optimization_problem_builder.problem_builder import build_problems_for_slab_type

# Build all problems for a slab type
bundles = build_problems_for_slab_type("solid_slab_one_way_concrete")

# Access a specific problem

# Extract components

# Get detailed results

# Returns: {"y": 150.5, "y_p": 152.3, "violations": {...}, "params": {...}}
```

## 2. Structure of `run_experiment()` Function

### Function Signature

```python
def run_experiment(
    problem_bundle: dict[str, dict],  # The bundles dictionary from problemBuilder()
    algorithm,                        # Optimizer object (callable with .name attribute)
    n_runs: int,                      # Number of independent runs per problem
    slab_type: str,                   # Slab type name (for folder naming)
    problem_ids: list[str] | None = None,  # Optional: specific problem IDs to run
) -> list[dict]:
```

### Function Flow

#### Step 1: Setup and Filtering
1. Creates output directory: `logger_results/` in current working directory
2. Generates timestamp for this experiment session
3. Filters bundles if `problem_ids` is provided:
   - If `problem_ids` is `None`: runs all problems in bundle
   - If `problem_ids` is a list: runs only those specific problems
4. Prints experiment configuration summary

#### Step 2: Iterate Through Selected Problems
For each problem in `selected_bundles`:

1. **Extract Bundle Components**:
   ```python
   p = b["problem"]                    # IOH problem
   ctx = b["ctx"]                      # Evaluation context
   var_names = b["var_names"]          # Variable names
   constraint_names = b["active_constraint_names"]  # Active constraints
   label = b.get("label", "")          # Problem label
   ```

2. **Prepare Context**:
   - `ctx.ensure_constraints(constraint_names)`: Declares constraint attributes
   - `ctx.ensure_params(var_names)`: Declares parameter attributes

3. **Create Logging Setup**:
   - Creates unique folder name: `{slab_type}__{problem_id}__{label}__{algorithm.name}__{timestamp}`
   - Initializes IOH logger (Analyzer) to track:
     - `y`: Unpenalized objective
     - `y_p`: Penalized objective
     - `misses`: Cache misses
     - `hits`: Cache hits
     - `c__<constraint_name>`: Each constraint violation
     - `var__<var_name>`: Each decoded parameter value
   - Attaches logger to problem

4. **Run Multiple Independent Runs**:
   ```python
   for run in range(n_runs):
       algorithm(p)                    # Run optimizer on problem
       run_best = p.state.current_best.y  # Get best value from this run
       ctx.reset()                     # Clear cache
       p.reset()                       # Reset problem state
   ```
   - Tracks best solution across all runs
   - Each run starts fresh (cache and problem state reset)

5. **Cleanup**:
   - Detaches logger from problem
   - Closes logger (writes data to disk)

#### Step 3: Summary and Return
- Collects results summary for all problems
- Prints summary table showing:
  - Problem ID
  - Label
  - Best objective value found
  - Output folder location
- Returns list of summary dictionaries:
  ```python
  [
      {
          "problem_id": "1",
          "label": "Height variation - fixed cost",
          "best_y": 152.34,
          "best_x": [2, 4, 1],
          "folder": "solid_slab_one_way_concrete__1__Height_variation__TPE__20250126_143022"
      },
      ...
  ]
  ```

### Algorithm Interface

The `algorithm` parameter must be a callable object with:
- `.name` attribute (string): Algorithm identifier (e.g., `"Hyperopt_TPE"`, `"NLOP_DIRECT"`)
- `__call__(problem)` method: Takes IOH problem, runs optimization, returns `(best_y, best_x)`

Example algorithm classes:
- `HyperoptAlgorithms`: Tree-structured Parzen Estimator
- `NloptAlgorithms`: DIRECT global optimization
- `RbfoptSearch`: Radial Basis Function optimization

### Output Structure

Results are saved in `logger_results/{folder_name}/`:
- IOH logger format files (`.dat` files)
- Contains per-evaluation data:
  - Objective values (`y`, `y_p`)
  - Constraint violations (`c__*`)
  - Parameter values (`var__*`)
  - Cache statistics (`hits`, `misses`)

### Example Usage

```python
from examples.run_penalized_demo import HyperoptAlgorithms, run_experiment
from slabdesignbench.optimization_problem_builder.problem_builder import build_problems_for_slab_type

# Build problems
bundles = build_problems_for_slab_type("solid_slab_one_way_concrete")

# Create algorithm
algo = HyperoptAlgorithms(hyperopt_algo="TPE", max_evaluations=50, seed=42)

# Run experiment
results = run_experiment(
    problem_bundle=bundles,
    algorithm=algo,
    n_runs=5,  # 5 independent runs per problem
    slab_type="solid_slab_one_way_concrete",
    problem_ids=["1", "2"]  # Optional: only run problems 1 and 2
)

# Access results
for res in results:
    print(f"Problem {res['problem_id']}: best = {res['best_y']}")
    print(f"  Best design: {res['best_x']}")
    print(f"  Results folder: {res['folder']}")
```

## Key Concepts

### Design Vector (`x_idx`)
- Integer index vector representing a design choice
- Each element is an index into a catalog/grid for that variable
- Example: `[2, 4, 1]` means:
  - Variable 0: index 2 (e.g., thickness = 200 mm)
  - Variable 1: index 4 (e.g., span = 5.0 m)
  - Variable 2: index 1 (e.g., reinforcement type B)

### Objective Values
- `y`: Unpenalized objective (raw cost/CO2/etc.)
- `y_p`: Penalized objective (includes constraint violations)
- Optimization minimizes `y_p`

### Constraint Violations
- Raw, non-negative values
- `0.0` = constraint satisfied
- `> 0.0` = constraint violated (magnitude indicates severity)

### Caching
- `EvalContext` caches analysis results to avoid redundant expensive computations
- Cache key: tuple of integer indices
- Cache hit: reuse previous result
- Cache miss: run analysis and store result

## 3. Algorithm Classes in `run_penalized_demo.py`

All algorithm classes follow the same interface: they are callable objects with a `.name` attribute and a `__call__(problem)` method.

| Class | Framework | Algorithm | Type |
|---|---|---|---|
| `PymooAlgorithms("GA", ...)` | pymoo | Genetic Algorithm | Populationsbasiert, global |
| `PymooAlgorithms("CMAES", ...)` | pymoo | CMA-ES | Evolutionaer, unimodal |
| `NloptAlgorithms("DIRECT", ...)` | NLopt | GN_DIRECT_L | Global, deterministisch |
| `NloptAlgorithms("SBPLX", ...)` | NLopt | LN_SBPLX | Lokal, Simplex |
| `HyperoptAlgorithms("TPE", ...)` | Hyperopt | Tree Parzen Estimator | Bayes'sch, sequenziell |
| `RbfoptSearch(...)` | RBFOpt | Radiale Basisfunktionen | Surrogate-basiert |
| `LatinHypercubeSampling(...)` | — | LHS | Baseline, kein Lernen |

### Gemeinsame Parameter
- `max_evaluations`: Maximales Auswertungsbudget (Anzahl Zielfunktionsauswertungen)
- `seed`: Zufallsseed fuer Reproduzierbarkeit (wird pro Run automatisch inkrementiert)

### Integer-Handling
Alle Algorithmen arbeiten intern mit reellen Zahlen. Die Konvertierung auf Integer erfolgt in der jeweiligen `__call__`-Methode via `int(round(xi))`, bevor das IOH-Problem aufgerufen wird. Ausnahme: `NloptAlgorithms` rundet in der lokal definierten `objective()`-Funktion.

### Externer Solver (RBFOpt)
`RbfoptSearch` benoetigt `bonmin.exe` und `libipoptfort.dll` im `solvers/`-Verzeichnis. Der Pfad wird automatisch relativ zur Skript-Position aufgeloest (`Path(__file__).parent.parent / "solvers"`).
